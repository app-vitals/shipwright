## Per-category detail

### 1. Pure business logic

The largest category by file count. Concentrated in:
- **`lib/`** — shared cross-service pure helpers (`pricing.ts`, `org-repo.ts`, `request-context.ts`, `secret-env-vars.ts`, `claim-ttl.ts`), imported by `agent/`, `admin/`, `metrics/`, `task-store/`. `claim-ttl.ts` is new since the last inventory and is now ranked critical — it defines the constant (`DEFAULT_CLAIM_TTL_MS`) that governs whether a stale-claim reaper fires too early mid-session, a recurring source of the "check-review stuck pending" class of bugs seen in prior incident history.
- **Decision/classification logic** — `agent/src/work-selector.ts` (FIFO candidate picker), `task-store/src/ready.ts` / `blocked-by.ts` / `statuses.ts` (dependency-satisfaction rules), `plugins/shipwright/scripts/classify_test_layer.ts`, `plugins/shipwright/scripts/check-helpers.ts`'s `isCleanApproveBody`.
- **Mutable-box refs** — `agent/src/agent-repos-ref.ts`, `agent/src/loop-jobs-ref.ts`: new pattern since the last inventory, both explicitly designed zero-I/O for unit-testability (per their own header comments) and give downstream cron/dispatch code a live view of async-synced state.
- **Pure wire-object builders** — `admin/src/agent-manifest.ts` (K8s Deployment/Secret shapes), `admin/src/dev-auth-guard.ts`.

### 2. Service-boundary code

Every workspace has a Prisma-backed service layer (task-store: `task-service.ts`/`pull-request-service.ts`/`token-service.ts`; chat: `message-service.ts`/`thread-service.ts`/`token-service.ts`; admin: `agent-envs.ts`/`agent-cron-jobs.ts`/`agent-chat-tokens.ts`/`agent-work-queue.ts`/etc.) plus internal HTTP clients (`metrics/src/lib/task-store-client.ts`, `agent/src/shipwright-*-client.ts`, `admin/src/*-provisioning-client.ts`). The agent's `check-*.ts` / `loop-orchestrator.ts` family is a distinctive service-boundary cluster: it composes GitHub API reads with task-store writes and is the operational core of the dev-task/review/patch/deploy loop — ranked critical; this cluster absorbed the largest share of net-new logic since the 2026-07-19 inventory (shared `isTaskBlockedForDispatch`/`isPrRecordBlockedForDispatch` helpers, PR-level `hitl`/`blockedBy` gating across all four `check-*.ts` files, `pr-state-reconciler.ts`'s new deploying→deployed reconciliation pass and throttled/repo-scoped GitHub calls, `loop-orchestrator.ts`'s redundant-patch-dispatch skip) but no new files and no category/layer change. `agent/src/chat-poller.ts` (introduced 2026-07-19) is service-boundary: it composes a chat-service client read/claim loop with the Claude runner, mirroring `loop-orchestrator.ts`'s shape but scoped to chat threads instead of task-store PRs. New this cycle: `mcp-server/src/http-transport.ts` (`mountMcpHttpTransport` — stateful Streamable HTTP session lifecycle with an idle-session reaper) and `mcp-server/src/auth.ts` (inbound bearer middleware, constant-time token compare) — both land as part of TSM-2.6/#1940/#1944/#1945's expansion of mcp-server from a stdio-only tool into a remotely-reachable HTTP service.

### 3. HTTP routes / public API

Every service (metrics, admin, task-store, chat, mcp-server) exposes a Hono app factory with a route table documented in its own header comment. `admin/src/server.ts` is a pure composition entrypoint — it re-exports and mounts the three existing admin sub-apps (`admin-ui.ts`, `agents-api.ts`, `api.ts`) with no new logic of its own, so it is classified alongside the other entrypoints rather than as a fourth distinct route surface. `agent/src/cron-handler.ts` and `agent/src/slack.ts` are treated as route-equivalent (own a wire contract with an external caller — the cron dispatcher and Slack respectively — even though they aren't URL-routed HTTP). mcp-server gained two new entrypoint-shaped files this cycle: `index.ts` (Hono app factory — mounts auth middleware + the MCP HTTP transport, composition-only like `admin/src/server.ts`) and `main.ts` (the `Bun.serve` process entrypoint — fails closed and refuses to start if `SHIPWRIGHT_MCP_SERVER_TOKEN` is unset, mirroring task-store's fail-closed auth posture). Canonical layer is smoke throughout, consistent with the CLAUDE.md convention (`*.smoke.test.ts` — in-process `app.request()`).

### 4. Error / failure paths

Thin category here — most error handling is folded into the service/route layer rather than living in dedicated retry/circuit-breaker modules (no queue/retry infrastructure observed). Distinct logic-only error modules: `metrics/src/lib/errors.ts`, `admin/src/errors.ts`, `task-store/src/errors.ts`, `chat/src/errors.ts` (typed HTTP error classes — unit), and `agent/src/cutover-validate.ts` (validation gate — unit). `task-store/src/stale-claim-reaper.ts` and `agent/src/pr-state-reconciler.ts` are boundary-involving error/self-heal paths — integration. `agent/src/cron-failure-reporter.ts` (new) is a boundary-involving error path: it catches a thrown cron-tick error, writes an AgentCronRun row via the admin API, and captures to Sentry — same shape as `pr-state-reconciler.ts`'s classification, ranked high given it is the only thing standing between a swallowed cron crash and operator visibility.

### 6. User journeys

Four top-level journeys identified (below the top-5 threshold the rubric anchors to — this is a smaller product surface than a typical consumer app, so 4 is the full set, not a truncated top-5). Unchanged from the last inventory, though journey #3's implementation detail is now more concrete with `chat-poller.ts` in place:

1. **dev-task → review → patch → deploy pipeline** — the core product loop (`/shipwright:dev-task` → `/shipwright:review`/`/shipwright:patch` → `/shipwright:deploy`), spanning `agent/`, `task-store/`, GitHub. Critical — this is the product.
2. **Agent provisioning** — admin creates an Agent record, which cascades through `agent-provisioner.ts` → K8s Deployment/Secret → task-store/chat-service token minting → the agent container boots and reports `/health`. Critical — this is how a customer/operator onboards a new agent.
3. **Slack chat with an agent** — DM/mention → chat-svc queue → `agent/src/chat-poller.ts`'s poll/claim loop → Claude session → Slack reply. High.
4. **Marketing site visitor journey** — landing → docs → self-hosted quickstart, already covered by `site/`'s existing Playwright `*.spec.ts` suite per CLAUDE.md. High (revenue/lead-gen path for an OSS project, not transactional revenue).

No journey is browser-driven except #4; #1-#3 are best proven by cross-service integration tests composing recorded fixtures (agent ↔ task-store ↔ GitHub-fixture) rather than a literal Playwright E2E, since there is no UI for the pipeline itself — noted as "e2e / smoke-composed" in the summary table to flag that the canonical E2E *tooling* (browser automation) doesn't apply, but the *layer* (multi-step, multi-service state) still does.
