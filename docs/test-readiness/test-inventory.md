## Sampling

- **Files inspected:** ~112 non-test source files sampled directly (headers/purpose + representative body), plus full directory enumeration (Glob) of all 8 workspaces and `plugins/shipwright/{commands,agents,skills,scripts,references}` (267 total non-test source files exist repo-wide; sampled ~42%, weighted toward files with non-obvious classification).
- **Sampling strategy:** breadth-first per workspace. For each of the 8 Bun workspaces plus `scripts/`, `brand/`, `site/src/`, and `plugins/shipwright/`, enumerated all non-test files via Glob, then read the doc-comment header (and first ~15-25 lines where the header was thin) of a representative subset covering every apparent purpose (route handlers, service classes, pure helpers, clients/wrappers, entrypoints). Existing test files were never read (per SKILL.md's "don't read test files for inventory"). Where multiple files shared an obvious pattern (e.g. `admin/src/*-provisioning-client.ts`, `task-store/src/routes/*.ts`, the ~20 `plugins/shipwright/skills/*/SKILL.md` prompt files), one or two were read in depth and the rest classified by filename + directory convention, noted as grouped rows below. The 22 SKILL.md prompt/skill files and ~30 command/agent `.md` files were classified as a single "content" grouping per the repo's own `*.content.test.ts` convention rather than read individually â€” reading each in full would have exceeded the sampling cap for low marginal signal (they are already covered by `plugins/shipwright/test/*.content.test.ts` and per-command `.content.test.ts` files).

## Per-category detail

### 1. Pure business logic

The largest category by file count. Concentrated in:
- **`lib/`** â€” shared cross-service pure helpers (`pricing.ts`, `org-repo.ts`, `request-context.ts`, `secret-env-vars.ts`), imported by `agent/`, `admin/`, `metrics/`.
- **Decision/classification logic** â€” `agent/src/work-selector.ts` (FIFO candidate picker), `task-store/src/ready.ts` / `blocked-by.ts` / `statuses.ts` (dependency-satisfaction rules), `plugins/shipwright/scripts/classify_test_layer.ts`, `plugins/shipwright/scripts/check-helpers.ts`'s `isCleanApproveBody`.
- **Pure wire-object builders** â€” `admin/src/agent-manifest.ts` (K8s Deployment/Secret shapes), `admin/src/dev-auth-guard.ts`.
- **Crypto helpers treated as pure** â€” `agent/src/crypto.ts`, `admin/src/crypto.ts`, `admin/src/token-crypto.ts`: deterministic given key/IV inputs, no ambient I/O; canonical layer is unit even though the category-classifier's "no I/O" signal is borderline (uses `node:crypto`, which is in-process, not a process boundary).
- **Parsing/validation utilities** â€” `scripts/check-coverage.ts` (lcov parsing), `scripts/check-config-docs.ts`, `brand/brand-lint.ts`, `brand/build-brand-css.ts`, `admin/src/attachment-validation.ts`.

These are already the repo's dominant unit-test surface (many have `*.unit.test.ts` siblings observed during enumeration, not read).

### 2. Service-boundary code

Every workspace has a Prisma-backed service layer (task-store: `task-service.ts`/`pull-request-service.ts`/`token-service.ts`; chat: `message-service.ts`/`thread-service.ts`/`token-service.ts`; admin: `agent-envs.ts`/`agent-cron-jobs.ts`/etc.) plus internal HTTP clients (`metrics/src/lib/task-store-client.ts`, `agent/src/shipwright-*-client.ts`, `admin/src/*-provisioning-client.ts`). The agent's `check-*.ts` / `loop-orchestrator.ts` family is a distinctive service-boundary cluster: it composes GitHub API reads with task-store writes and is the operational core of the dev-task/review/patch/deploy loop â€” ranked critical.

### 3. HTTP routes / public API

Every service (metrics, admin, task-store, chat, mcp-server) exposes a Hono app factory with a route table documented in its own header comment. `agent/src/cron-handler.ts` and `agent/src/slack.ts` are treated as route-equivalent (own a wire contract with an external caller â€” the cron dispatcher and Slack respectively â€” even though they aren't URL-routed HTTP). Canonical layer is smoke throughout, consistent with the CLAUDE.md convention (`*.smoke.test.ts` â€” in-process `app.request()`).

### 4. Error / failure paths

Thin category here â€” most error handling is folded into the service/route layer rather than living in dedicated retry/circuit-breaker modules (no queue/retry infrastructure observed). Distinct logic-only error modules: `metrics/src/lib/errors.ts`, `admin/src/errors.ts`, `task-store/src/errors.ts`, `chat/src/errors.ts` (typed HTTP error classes â€” unit), and `agent/src/cutover-validate.ts` (validation gate â€” unit). `task-store/src/stale-claim-reaper.ts` and `agent/src/pr-state-reconciler.ts` are boundary-involving error/self-heal paths â€” integration.

### 5. External integrations

- `admin/src/kubernetes-client.ts` â€” in-cluster K8s API (critical; already has a `RecordedKubernetesClient` cassette double per its own header comment).
- `admin/src/google-auth-client.ts` â€” Google OAuth2 (critical, auth path).
- `agent/src/github-app-auth.ts` â€” GitHub App installation tokens (critical).
- `agent/src/voice.ts` â€” Groq Whisper / ElevenLabs (medium).
- `admin/src/slack-provisioning-client.ts` â€” Slack `apps.manifest.create` (high, one-time provisioning flow).
- `lib/sentry.ts` â€” Sentry SDK wiring (high, observability-critical but not user-facing).
- Third-party SDKs used directly in entrypoints (`@sentry/bun`, `@octokit/*`, `@slack/bolt`) are not separately inventoried beyond the wrapper modules that own them.

### 6. User journeys

Four top-level journeys identified (below the top-5 threshold the rubric anchors to â€” this is a smaller product surface than a typical consumer app, so 4 is the full set, not a truncated top-5):

1. **dev-task â†’ review â†’ patch â†’ deploy pipeline** â€” the core product loop (`/shipwright:dev-task` â†’ `/shipwright:review`/`/shipwright:patch` â†’ `/shipwright:deploy`), spanning `agent/`, `task-store/`, GitHub. Critical â€” this is the product.
2. **Agent provisioning** â€” admin creates an Agent record, which cascades through `agent-provisioner.ts` â†’ K8s Deployment/Secret â†’ task-store/chat-service token minting â†’ the agent container boots and reports `/health`. Critical â€” this is how a customer/operator onboards a new agent.
3. **Slack chat with an agent** â€” DM/mention â†’ `chat-poller.ts`/chat-svc queue â†’ Claude session â†’ Slack reply. High.
4. **Marketing site visitor journey** â€” landing â†’ docs â†’ self-hosted quickstart, already covered by `site/`'s existing Playwright `*.spec.ts` suite per CLAUDE.md. High (revenue/lead-gen path for an OSS project, not transactional revenue).

No journey is browser-driven except #4; #1-#3 are best proven by cross-service integration tests composing recorded fixtures (agent â†” task-store â†” GitHub-fixture) rather than a literal Playwright E2E, since there is no UI for the pipeline itself â€” noted as "e2e / smoke-composed" in the summary table to flag that the canonical E2E *tooling* (browser automation) doesn't apply, but the *layer* (multi-step, multi-service state) still does.
