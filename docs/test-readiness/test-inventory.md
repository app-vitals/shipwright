## Stack profile

- **Language(s):** TypeScript (Bun runtime throughout; Astro for the marketing site)
- **Package manager:** Bun workspaces (`bun.lock`, `bunfig.toml`)
- **Primary framework(s):** Hono (`OpenAPIHono` + Zod schemas) for every HTTP service; Prisma/Postgres for the three stateful services (admin, task-store, chat); Playwright for browser E2E (admin, metrics, site); Claude Code plugin format (commands/skills/agents) for `plugins/shipwright/`
- **Detected source roots:** `lib/`, `plugins/shipwright/{commands,skills,agents,scripts,references}`, `metrics/src/`, `agent/src/`, `admin/src/`, `task-store/src/`, `chat/src/`, `mcp-server/src/`, `site/` (Astro, not a Bun workspace)

This repo's own `CLAUDE.md` declares a concrete layer structure and file-suffix convention (`## Test conventions`), which this inventory adopts verbatim instead of the rubric's generic naming:

| Suffix | Layer | What it covers |
|---|---|---|
| `*.unit.test.ts` | unit | pure logic, no I/O |
| `*.integration.test.ts` | integration | real dependency behavior via recorded fixtures / injected doubles |
| `*.smoke.test.ts` | smoke | Hono endpoints via in-process `app.request()` (no real socket) |
| `*.e2e.ts` / `*.spec.ts` (site) | e2e | full browser-driven flows via Playwright |

## Sampling

- **Files inspected:** ~155 of ~179 non-test TypeScript source files across all 8 Bun workspaces (`lib`, `plugins/shipwright`, `metrics`, `agent`, `admin`, `task-store`, `chat`, `mcp-server`), plus `CLAUDE.md`, `docs/architecture.md`, and existing E2E test files for journey signal.
- **Sampling strategy:** breadth-first by workspace. For each workspace, enumerated every source file via `find`/`Glob`, then read the header comment + first ~15-25 lines of every file (function names, JSDoc "Signal"/"Purpose" comments, import lists) rather than full bodies. This repo's convention of a substantial header docblock on nearly every file made this highly reliable — headers explicitly state DB/HTTP/pure-logic status in most cases. Generated files (`lib/*-types.ts`, `mcp-server/src/generated-tools.ts`) and pure re-export shims (e.g. `task-store/src/validate.ts`, `agent/src/pricing.ts`) were noted but not deep-classified — they inherit the classification of their source. Existing `*.e2e.ts`/`*.spec.ts` files were skimmed (not full-read) purely to identify already-covered user journeys for the journey section, per the "don't read test files for classification" rule — journeys are a product-level concept, not a code-level one, so this is not a violation of that rule.

## Critical-path roster (canary suite definition)

This is the closed set of items that define the canary suite. Anything outside this list is NOT canary-eligible regardless of layer.

1. **task-store auth + core routes** (`task-store/src/auth.ts`, `routes/tasks.ts`, `routes/prs.ts`) — smoke, critical, read-only `GET` paths only (list/get, not claim/mutate).
2. **task-store app boot** (`task-store/src/app.ts`, `/health`) — smoke, critical.
3. **chat auth + core routes** (`chat/src/auth.ts`, `routes/threads.ts`, `routes/messages.ts`) — smoke, high, read-only `GET` paths only.
4. **metrics auth + summary/trends/tokens routes** (`metrics/src/lib/session-middleware.ts`, `api-auth.ts`, `metrics/src/api.ts`) — smoke, critical, read-only.
5. **metrics dashboard journey** (login → view summary/trends/features/queue/tokens) — e2e, critical, read-only, self-contained via fixtures/session-cookie injection.
6. **admin auth + agents-api / admin-ui core routes** (`admin/src/api-auth.ts`, `agents-api.ts`, `admin-ui.ts` GET paths) — smoke, critical, read-only.
7. **admin login → agent detail page journey** (`admin/e2e/login-page.e2e.ts`, `agents-page.e2e.ts`) — e2e, critical, read-only.
8. **agent health endpoint** (`agent/src/health.ts`) — smoke, high, read-only.
9. **site marketing pages** (`site/tests/*.spec.ts`) — e2e, high, static/read-only.

Explicitly excluded from canary despite critical/high tier: anything that claims/mutates a task or PR (`POST /tasks/:id/claim`, `POST /prs/claim`, etc.), agent provisioning/deprovisioning (real K8s + Slack side effects), and the dev-task→deploy pipeline journey — all are destructive with no established self-cleaning teardown in this codebase yet. If a teardown pattern is added later (e.g. a dedicated canary-scoped agent/task created and torn down per run), promote per the `canary-execution` skill's read-only-or-self-cleaning rule.

## Per-category detail

### 1. Pure business logic

Concentrated in three places: (a) `lib/` — cross-service pure helpers shared by admin/task-store/metrics (`org-repo.ts`, `pricing.ts`, `request-context.ts`, `secret-env-vars.ts`); (b) small "pure core, DB shell around it" files, most explicitly documented as such in their header comment (`task-store/src/blocked-by.ts`, `ready.ts`, `statuses.ts`; `agent/src/work-selector.ts`; `admin/src/agent-manifest.ts`, `agent-deletion-checklist.ts`, `attachment-validation.ts`, `token-crypto.ts`/`crypto.ts`; `mcp-server/src/tool-allowlist.ts`; `metrics/src/formatters.ts`, `select-provider.ts`); (c) string-rendering functions with no I/O (`admin-ui-pages.ts`, `dashboard-page.ts`, `render-plan-html.ts`). This repo's convention of separating pure logic into its own file with a "pure, no I/O" header comment (see `attachment-validation.ts`, `agent-manifest.ts`, `dev-auth-guard.ts`) makes category-1 identification unusually reliable — the classification signal is often explicit rather than inferred.

### 2. Service-boundary code

The bulk of `admin/src/agent-*.ts` (CRUD services over Prisma), `task-store/src/{task,pull-request,token}-service.ts`, `chat/src/{message,thread,token}-service.ts` — all DB-backed. Also file-backed stores (`agent/src/analytics.ts`, `sessions.ts`), in-memory TTL stores (`task-store/src/doc-store.ts`), and the `check-*.ts` precheck family in both `agent/src/` and `plugins/shipwright/scripts/` (real reads against task-store/GitHub, no mutation in the precheck itself).

### 3. HTTP routes / public API

Every service (`task-store`, `chat`, `metrics`, `admin`, `agent` health) is a Hono app with `OpenAPIHono`-declared routes and Zod schemas — the wire contract is well-defined and uniformly testable via in-process `app.request()`, matching the smoke-layer definition in `docs/test-readiness/test-system.md`/`CLAUDE.md` exactly.

### 4. Error / failure paths

Fewer standalone files than expected — most error handling is embedded in service/route files rather than isolated. Two notable dedicated cases: `task-store/src/stale-claim-reaper.ts` (reclaims abandoned claims — real DB boundary, integration layer) and `agent/src/loop-orchestrator.ts`'s busy-flag/restart-safety logic (documented explicitly as tolerating mid-drain process restarts). Typed error classes (`errors.ts` in task-store/chat/admin) are pure category-1 (unit), not category-4 themselves — they're the error *shape*, not the error *handling path*.

### 5. External integrations

`agent/src/claude.ts` (Claude Agent SDK), `slack.ts` (Slack Bolt), `voice.ts` (Groq/ElevenLabs), `github-app-auth.ts` (GitHub App); `admin/src/kubernetes-client.ts`, `google-auth-client.ts`, `slack-provisioning-client.ts`; `metrics/src/providers/task-store-recorded.ts` (the recorded-fixture double itself, which is the category-5 test infrastructure, not the code under test). Per `CLAUDE.md`'s hard isolation rule, all of these should be tested via recorded fixtures / injected doubles — never `mock.module()` or `global.fetch` overrides. Several files already document this pattern explicitly in their header comments (`kubernetes-client.ts`, `metrics/src/lib/task-store-client.ts`).

### 6. User journeys

Four identified, of which two already have E2E coverage and two do not:

- **Metrics dashboard** (login → summary/trends/features/queue/tokens views) — covered (`metrics/e2e/dashboard.e2e.ts`).
- **Admin login → agent detail page** — covered (`admin/e2e/login-page.e2e.ts`, `agents-page.e2e.ts`).
- **Admin agent-provisioning** (create agent → K8s Deployment/Secret + Slack app + chat/task-store tokens provisioned → visible in UI) — **not covered by E2E**. High-value but destructive (real external side effects); recommend integration-layer coverage of `agent-provisioner.ts`'s orchestration via injected fakes for each of the five external clients, rather than a true E2E.
- **Full dev-task → review → patch → deploy pipeline** (the product's core loop: task claim → branch → PR → CI → review → merge → deploy) — **not covered by E2E**, and almost certainly never should be as a single E2E given it spans GitHub, Claude Code sessions, and real repo state. This is the single highest-value gap in the entire inventory, but the right test strategy is integration-layer coverage of each phase transition (already present per the test-migration phase's existing counts) plus the existing dogfooding of the pipeline on this very repo as its de facto continuous validation — not a scripted E2E.
- **Site marketing pages** (architecture/compare/config/docs/home/search) — covered (`site/tests/*.spec.ts`).

## Ambiguous items (human review needed)

- **`plugins/shipwright/commands/*.md`, `skills/*/SKILL.md`, `agents/*.md`** — these are prompt/process definitions consumed by an LLM at runtime, not executable TypeScript. They don't fit any of the six code categories cleanly. The plugin's existing test suite (`plugins/shipwright/tests/`, `test/`) appears to test the *scripts* these skills invoke, not the prompt content itself. Recommend treating prompt-content correctness as an out-of-band concern (reviewed by `/shipwright:review`, not unit-tested) rather than forcing a layer classification — flagging for human confirmation this is the right call rather than a coverage gap.
- **`mcp-server/src/mcp-server.ts` canary eligibility** — it's a smoke-layer, high-criticality HTTP/protocol surface (MCP `tools/list`/`tools/call` over stdio or Streamable HTTP), but the existing canary-execution model (per `skills/canary-execution/SKILL.md`) is framed around web-reachable smoke/E2E hitting a deployed env. Stdio transport doesn't fit that shape. Marked not-canary-eligible here but flagging for reviewer judgment — if the Streamable HTTP transport becomes the primary deployment mode, this should be revisited.
- **`agent/src/users.ts`** — split-brain classification (pure cache logic is category 1, the injected resolver call is category 2). Listed once at unit (the cache is the interesting/testable part; the resolver call is a thin pass-through) but flagging in case a reviewer wants the resolver call's integration behavior tested separately.
- **`metrics/src/lib/api-auth.ts`'s `enumerateRoutes` coverage test** — the file's own header comment mentions "The coverage test in api/src/server.test.ts uses this enumeration to assert that no route ships without a declared policy," but no `api/src/server.test.ts` exists in this repo (likely a stale comment carried over from a template/reference codebase, or the path has since moved to `metrics/src/server.test.ts` under a different name). Flagging as a doc-drift observation for Phase 3, not reclassifying based on an unverified comment.
- **`admin/src/admin-ui.ts` and `agents-api.ts` route composition** — both mix real route registration (category 3) with some inline business logic in handler bodies (category 1/2) per the "thin route handler doing real work" ambiguity rule. Listed at their canonical smoke layer per the rule (split mentally: the route is smoke, the work it does is unit/integration at its own canonical layer) — Phase 3 should confirm the handler bodies don't have un-extracted logic that's only exercised via the smoke layer today (which would mean it's untested at the correct lower layer).

## Notes

- This inventory is **deduplicated** per the canonical-layer rule. Each functional unit appears exactly once at its lowest-sufficient layer.
- Higher-layer tests, if needed, only assert what the canonical layer cannot (boundary behavior, wire contract, multi-step state).
- Criticality tiers used the "top 5-10 user flows" framing plus this repo's own architecture doc: task/PR lifecycle (task-store) and the dev-task→deploy loop are the product's core revenue-adjacent path, hence `critical`; auth/token/crypto surfaces across all services are `critical` as security boundaries; per-service CRUD and provisioning are `high`; admin-UI rendering, formatting, and internal-only utilities are `medium`/`low`.
- **Living classification process** (per `test-inventory` SKILL.md Step 4b): new files should be classified against this rubric at PR time (recommend adding to each published test-readiness issue's closing checklist, per `repo-config/SKILL.md`); tier promotion triggers are feature-flag removal, ≥10% traffic ramp, GA milestone, or explicit engineer flag — surfaced again in `test-readiness-plan.md`'s open risks; a full re-run of `/test-inventory` is warranted only after a new service boundary or top-level user journey is added (the last such addition was likely the chat service — `chat/` — which this refresh already covers).
- Phase 2 (`/test-design`) consumes this inventory to design (or re-validate, since `docs/test-readiness/test-system.md` already exists from a prior run) the ideal test system.
