## Method note

This repo has **~220 test files (~130K lines including source) / ~110K lines of test
code** across 8 workspaces plus `scripts/`, `brand/`, `plugins/shipwright/`, and `site/`.
Reading every file in full was not feasible within this pass. The approach taken:

1. Enumerated every test file via the Phase 2 framework matrix's glob patterns (`bun
   test`'s default discovery for `*.unit.test.ts` / `*.integration.test.ts` /
   `*.smoke.test.ts` / `*.content.test.ts`, plus `**/*.spec.ts` for `site/`, plus
   `**/e2e/**/*.e2e.ts` for `admin/`/`metrics/` Playwright).
2. Measured every file's line count as a first-pass layer-mismatch signal (a "unit" test
   with real DB/process I/O tends to also be disproportionately large or complain in its
   own header comment).
3. Read file headers + representative bodies for every workspace cluster, weighted toward
   files whose name/inventory pairing looked ambiguous (e.g. `*.integration.` on a file
   with no I/O; `*.unit.` on a file wrapping an external client).
4. Grepped the **entire** test suite for the isolation contract's hard signals
   (`mock.module(`, `global.fetch =`, `global.console =`, `DATABASE_URL` inside
   `*.unit.test.ts`) to find real violations vs. false positives (comments documenting
   compliance).
5. Cross-referenced every Phase 1 inventory row against the test-file glob to find
   zero-coverage items.
6. Checked `git log --follow` for every `site/tests/*.spec.ts` and `*/e2e/*.e2e.ts` file
   for outage-linked commits before considering any trim.

**Speed measurement: not run.** No local Postgres is available in this environment
(`DATABASE_URL_SHIPWRIGHT_*` unset, `pg_isready` absent) — running `bun test` here would
silently skip every DB-backed integration test and produce misleading "fast" timings for
files that actually hit real Postgres in CI. Per SKILL Step 4 ("optional but
recommended"), this was skipped rather than produce a misleading measurement; line-count
and import-based static signals were used instead as the layer-mismatch heuristic. CI's
own historical full-suite timings (referenced in Phase 2's Speed budgets section: "low
minutes" wall time, comfortably under the 15-minute budget) are the source of truth for
actual timing — no evidence found of any test individually breaching a hard cap.

**What this pass did *not* do:** open all ~220 files in full. For the largest clusters
(admin, agent, task-store — each with 15-40 test files), every file's header comment and
declared strategy were read, and a sample of full bodies per functional group (auth,
CRUD service, HTTP route, client wrapper) was read in depth. No file was bucketed on
filename/directory alone — every bucket assignment below is backed by an actual read,
even if abbreviated for very large or very repetitive files.

## Trim analysis

**Zero trims identified.** The SKILL's trim bucket requires a *specific, cited* duplicate
assertion — a higher-layer test re-asserting a business rule already proved by a
lower-layer test for the same module. Sampling across every workspace found the opposite
pattern consistently: each layer's test file asserts a genuinely distinct concern for its
module (e.g. `admin/src/admin-ui-pages.unit.test.ts` asserts pure render-function output;
its sibling `admin/src/admin-ui.smoke.test.ts` asserts the routed HTTP wrapper around those
same render functions — different concern, not duplicate coverage). Two candidate pairs
were checked in depth and found non-duplicative:

- `plugins/shipwright/commands/dev-task.content.test.ts` (markdown-prose assertions about
  the `/dev-task` command's Step 1 orphan-check behavior) vs.
  `plugins/shipwright/commands/dev-task.unit.test.ts` (pure parsing/scoring logic in a
  colocated helper) — no assertion overlap, different subjects entirely despite the shared
  filename stem.
- `plugins/shipwright/agents/learning-dreamer.integration.test.ts` vs.
  `plugins/shipwright/agents/learning-dreamer.unit.test.ts` — both read the same
  `learning-dreamer.md`, but each extracts and asserts on a different section (mining-table
  vs. review-mode); no duplicate assertion found.

No E2E test (`site/tests/*.spec.ts`, `admin/e2e/*.e2e.ts`, `metrics/e2e/*.e2e.ts`) was
considered for trimming — per the SKILL's hard rule, E2E tests are never deleted, and none
were found re-asserting only what a lower layer already proves (all 12 site specs assert
page-level rendering/navigation, which by definition is not provable at unit/integration
layer). All 15 E2E-layer files' git history was checked (`git log --follow`); none carry
an outage-linked commit, but that only matters for E2E *retention* decisions (per SKILL:
"if it has an outage-linked commit, mark it reuse and leave it") — since no E2E file was a
trim/delete candidate in the first place, this check served as confirmation, not a
deciding factor.
