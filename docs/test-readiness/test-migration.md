## Method note

211 existing test files were discovered via `find . -type f \( -name "*.test.ts" -o -name "*.e2e.ts" -o -name "*.spec.ts" \)`. Given the volume, this pass bucketed at the **file-group level** (files sharing a functional unit, module, or naming pattern) rather than reading all 211 files individually — every file's header docblock (present on the large majority) and import list were read; ~35 files were read in full for verification of ambiguous naming. The actual suite was run (`bun test --filter unit`, `--filter smoke`, `--filter integration`) to get real pass/fail and timing data rather than estimating — see Speed measurement below. E2E suites (Playwright) were not executed — this sandbox lacks a launchable Chromium (a known, pre-existing environment gap, not evidence about the tests themselves) — E2E bucketing below is based on file/spec inspection only.

**Note on `task-store/src/{pull-request-service,stale-claim-reaper}.unit.test.ts` and `admin`/`chat` service-file `.unit.test.ts` siblings:** Phase 1's inventory prescribed **integration** as the canonical layer for service-boundary code with real DB behavior, and for error paths involving a real boundary (`stale-claim-reaper.ts` specifically). In practice, this repo's actual convention — verified by reading these files — is to test the pure decision logic of these services via a hand-built Prisma **double** (not a real DB) at the unit layer, and reserve the real-DB `.integration.test.ts` sibling file for the boundary behavior itself (schema compatibility, transaction semantics, concurrent-write races). This is a **legitimate and correct** application of the canonical-layer rule, not a deviation: the double-backed unit test proves the decision logic (e.g. "409 on same commitSha with reviewState !== pending"), while the real-DB integration test proves the boundary actually persists/enforces that same logic — no functional overlap, each layer proves what the other cannot. Phase 2's blueprint should be read as compatible with this pattern (it already lists `Recorded*Client`/injected-double integration tests as the norm; a hand-built Prisma double used for pure-decision-logic unit tests is the DB-analogue of the same DI discipline). No migration action needed — noting explicitly so this isn't mistaken for a layer violation in Phase 4.
