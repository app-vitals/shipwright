## Method note

This is the fourth `/test-migration` pass (prior: 2026-07-16, 2026-07-19, committed as
this file's history shows). Rather than re-read all ~243 test files cold, this pass
diffed the current file tree against the 2026-07-19 pass's findings and re-verified each
open item directly, then did a fresh full-suite discovery pass to catch anything the diff
approach would miss:

1. Enumerated every test file via the Phase 2 framework matrix's glob patterns (`bun
   test`'s default discovery for `*.unit.test.ts` / `*.integration.test.ts` /
   `*.smoke.test.ts` / `*.content.test.ts`, plus `site/**/*.spec.ts`, plus
   `{admin,metrics}/e2e/**/*.e2e.ts`).
2. Re-checked every open item from the 2026-07-19 report (1 net-new test â€”
   `admin/src/clock.ts`, low-priority/optional; 0 net-new infra) against the current tree
   by direct file-existence checks â€” not assumed from memory.
3. Cross-referenced Phase 1's "new since the 2026-07-19 inventory" file list (7 new
   non-test source files, concentrated in `mcp-server/` + `metrics/src/lib/coalescing-cache.ts`)
   against the test-file glob to find newly-introduced coverage gaps.
4. Re-ran the isolation-contract grep sweep (`mock.module(`, `global.fetch =`,
   `global.console =`, `DATABASE_URL` inside `*.unit.test.ts`) across the full current
   suite, not just the delta, to catch any regression introduced by unrelated PRs since
   the last pass.
5. Confirmed `.github/PULL_REQUEST_TEMPLATE.md` and `bunfig.toml`'s `pathIgnorePatterns`
   directly rather than trusting the prior doc's snapshot.
6. Read `mcp-server/src/http-transport.smoke.test.ts` in full to confirm the idle-session
   reaper's timing logic (flagged as a watch item in this pass's own Phase 1/2 docs) is
   actually exercised via injected `Clock` advancement, not left untested inside a
   composition-only smoke file.

**Speed measurement: not re-run this pass**, same reasoning as all three prior passes â€” no
local Postgres in this sandbox (`DATABASE_URL_SHIPWRIGHT_*` unset), so a `bun test` run here
would silently skip every DB-backed integration test and produce misleading timings.
Static review (line counts, import/mock shape) was used instead; no evidence found of any
test breaching a hard cap. A real per-layer measurement continues to exist at
`docs/test-readiness/speed-baseline.md` on `main` (unchanged since 2026-07-15, already
cross-referenced into `test-system.md`'s Speed budgets section since the 2026-07-19 pass
per T-067/PR #1930 â€” no further correction needed this cycle). See
`project_shipwright_metrics_full_suite_flake` and `project_shipwright_admin_full_suite_flake`
in memory for the two known pre-existing full-suite flakes (unrelated to layer/speed, not
relevant to this audit).
