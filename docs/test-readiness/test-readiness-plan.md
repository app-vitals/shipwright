## 6. Open risks

- **Process gap in Phase 3's existence checks (human call on whether to fix now or track as a learning).** Phase 3's carried-forward "still open" verdict on the PR template was produced by a case-sensitive `ls .github/pull_request_template.md`, which missed the real file at `.github/PULL_REQUEST_TEMPLATE.md`. This is the second corrected claim this cycle (the other being the speed-measurement escalation) — both were catchable by checking task-store status and the filesystem directly rather than trusting carried-forward prose. Recommend a durable fix to `test-migration/SKILL.md`'s existence-check guidance (case-insensitive glob, not a literal exact-case `ls`) — flagged here as a `# Harness TODO` candidate for `test-migration/SKILL.md` rather than actioned in this doc-only cycle, since it's a skill-process fix, not a repo gap.
  ```
  # Harness TODO: test-migration/SKILL.md's existence-check steps (e.g. verifying
  # .github/pull_request_template.md) should use a case-insensitive glob rather than an
  # exact-case `ls`/path check — GitHub recognizes both `pull_request_template.md` and
  # `PULL_REQUEST_TEMPLATE.md` as the canonical template path, and a literal-case check
  # can produce a false "still open" carry-forward across cycles.
  ```
- **Speed baseline is a point-in-time measurement (2026-07-15).** It predates this cycle's ~17-file delta. Nothing in that delta is large enough to plausibly threaten the <15min full-PR budget (per `test-system.md`'s own override-condition analysis), but the baseline itself doesn't reflect the current file count. Not urgent enough to re-measure this cycle; worth a fresh capture the next time suite growth is non-trivial or a flake/slowness pattern is reported.
- **Recorded-fixture coverage breadth** (Phase 2's gaps-summary item 2: ~9 remaining `Http*Client` modules that could extend the `Recorded*Client` pattern) is a design-level backlog item, not a migration-bucket verdict on any existing file — no task emitted this cycle since Phase 3 found no test currently missing coverage because of it. Revisit if a future inventory pass tags one of those clients as critical/high with no test.
- **`agent/src/piper-voice.ts` reclassification watch item** (Phase 1/2: unit today, integration once Piper synthesis lands per PPR-1.2) — no action needed until that lands; flagged so a future Phase 3 doesn't silently miss the transition.
- **Recorded-fixture maintenance loop remains deferred** (`repo-config/SKILL.md` §5) — correctly not scheduled this cycle; sequence after fixture-authoring work matures, per Phase 2's explicit guidance.
- **T-051 (branch-protection required-checks verification) is not resolved — carried forward as T-071, not dropped.** The prior cycle's task list tracked `test-t-051-shipwright` (verify `main`'s live branch-protection required-checks list against `ci.yml`'s job names) as `pending — still open`.
