## 4. Roadmap (five milestones)

### Milestone 1 — Infrastructure baseline
**DOD:** Runners, local substitutes, CI pipeline shape in place. No new tests yet — just the rails.

The naming convention and runner-exclusion config (the SKILL's mandatory M1 item) already
exist and are correct — `docs/test-readiness/naming.md` and `bunfig.toml`'s
`pathIgnorePatterns` (`**/site/**`, `**/metrics/e2e/**`, `**/admin/e2e/**`,
`**/*.canary.test.ts`) match exactly, confirmed by direct read. T-048 below is therefore a
**verification** task (confirm + close the loop with a one-time isolation check), not a
creation task — the SKILL still requires this task to exist explicitly per its "must
always include" language, even when the target state is already met, so the verification
is captured as a real, closeable unit of work rather than silently assumed.

Remaining M1 work: wire chat's Postgres DB into CI (a step inside the existing,
already-required `ci` job — no new job name, so **no paired branch-protection task** per
the pairing rule's own scope: the rule triggers on a new required-check *name*, and
adding a step to an already-required job doesn't change what's required — see task note),
add the missing PR template, and verify the live branch-protection required-checks list
matches `ci.yml`'s actual job names (a precondition for the pairing rule elsewhere in this
plan being enforceable at all).

## 6. Open risks

- **Speed measurement gap.** No per-layer p95 was captured this cycle (no local Postgres
  in the Phase 3 environment). The Speed delta table above is unfilled. Recommend the
  next `/test-migration` run happen in an environment with `DATABASE_URL_SHIPWRIGHT_*`
  set and Postgres reachable so real numbers replace "not measured."
- **Branch-protection required-checks list not yet verified against the live GitHub API**
  (T-051 exists to close this, but until it runs, the pairing-rule reasoning in T-049's
  note — "the `ci` job is already required, so no new branch-protection task is needed" —
  rests on an assumption carried from Phase 2, not a confirmed API read). If `gh api
  .../protection` shows `ci` is *not* currently required, T-051 becomes a real
  branch-protection change, not a no-op verification.
- **Recorded-fixture-breadth ground-truth correction.** This phase found Phase 2's ~12-file
  estimate for "remaining `Http*Client` needing recorded-fixture doubles" was ~40% too
  high (7 of 12 already covered). This is not a risk to the plan above (the 5 real gaps
  are individually verified), but it is a signal that Phase 2's greenfield estimates
  should be treated as upper bounds requiring Phase 4 ground-truth verification before
  becoming tasks — which is exactly what happened here, but is worth flagging so a future
  cycle doesn't skip the verification step and over-schedule already-done work.
- **Chat DB CI-wiring sequencing.** T-054/T-055/T-056 `depends_on: T-049` is a real
  technical dependency (the tests need `shipwright_chat_test` to exist to pass in CI, not
  just locally against a developer's own Postgres). If T-049 is deferred or reverted,
  these three tasks cannot land independently — they would fail CI even if the test code
  itself is correct.
- **Milestone 3 (canary) permanently empty under the current deploy model.** Not a risk to
  this cycle, but worth a standing note: if `app-vitals/shipwright` ever moves to a staged
  deploy model (staging → canary → promote), the next `/test-roadmap` run needs to
  activate Milestone 3 using the 4-step checklist already captured in `test-system.md`'s
  Canary execution contract section — this plan does not need to be re-derived from
  scratch at that point.
