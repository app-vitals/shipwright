## 3. The gap

The gap in this repo is unusually narrow — most of what a greenfield Phase 2 design would prescribe is **already built and already matches**, because this pipeline has run here before (`test-system.md` predates this refresh) and the team has evidently been building to its own documented convention. The actionable gap decomposes into five concrete items, none of which requires new framework adoption or architectural change:

1. **A real, narrow test-isolation bug class (4 files, 12 cases):** three distinct root causes (ambient plugin-marketplace directory, ambient Kubernetes ServiceAccount CA file, ambient `ANTHROPIC_MODEL` env var) all manifesting as "test reads real environment state instead of an injected value." This is a correctness gap, not a missing-coverage gap — the assertions are right, the isolation is wrong.
2. **Canary suite is fully unwired** (0 of 9 target items have `TEST_TARGET_URL` plumbing) despite the underlying tests already existing in a canary-compatible shape. Pure infrastructure task.
3. **Chat service integration tests are structurally invisible in CI** — no DB service-container wiring exists for `DATABASE_URL_SHIPWRIGHT_CHAT_TEST`, so even if someone writes the first chat `.integration.test.ts` today, it will silently skip forever in CI (matching this agent's own prior observed pattern of CI-invisible skip conditions across other Shipwright services). Zero current impact (chat has 0 integration tests today) but a landmine for the first one written.
4. **One real coverage hole:** the admin agent-provisioning journey (5-way composition across K8s/Slack/chat/task-store/DB) has per-client integration coverage but no composition-level test proving the whole orchestration in `agent-provisioner.ts`/`agent-deletion.ts` works end-to-end (order-of-operations, partial-failure retry semantics documented in the source but not exercised together).
5. **A test-taxonomy naming gap**, not a coverage gap: ~15 files (`plugins/shipwright/commands/*.test.ts`, `plugins/shipwright/skills/{investigate-cron,learning-capture}/SKILL.integration.test.ts`, a few `scripts/check-*.test.ts` unsuffixed siblings) test markdown prompt/skill content rather than executable TypeScript, and don't fit the four-layer suffix convention cleanly. These tests pass and are valuable; the gap is purely a documentation/naming-consistency one.

No missing layers, no wrong-framework tests, no external dependency lacking a local substitute, and (per Phase 3) no confirmed redundant/trimmable assertions.

## 4. Roadmap (five milestones)

### Milestone 1 — Infrastructure baseline
**DOD:** Runners, local substitutes, CI pipeline shape in place. No new tests yet — just the rails.

- Naming convention is **already established and already followed** (`CLAUDE.md`'s `## Test conventions` table) — no net-new convention-definition task needed, but the content-assertion-test taxonomy gap (item 5 in The Gap) is the one loose end and is sequenced here as a documentation task (R2-001).
- Fix the four isolation-bug test files (R2-002 through R2-005) — these are infra-adjacent (test hygiene) and gate cleanly before any new coverage is added, since a flaky/environment-sensitive test undermines trust in everything downstream.
- Wire chat's Postgres service container into CI (R2P-006 / R2-006a / R2-006b) — pure infra, unblocks Milestone 2/4's chat integration net-new work without landmining it.

### Milestone 2 — Critical-path coverage
**DOD:** 100% of `critical` tier inventory items have passing tests at the prescribed layer.

- Per Phase 1's roster, all 9 critical-path items already have passing tests at the prescribed layer (verified live in Phase 3) — **this milestone's DOD is already met** for existing critical-tier items. The one exception is the admin agent-provisioning journey (critical tier, currently only per-client integration coverage, no composition test) — R2-007.

### Milestone 3 — Canary suite live
**DOD:** Smoke + E2E canary-eligible tests run green against the deployed env in <60s wall time.

- R2-008: build the `TEST_TARGET_URL` mode-switch helper (shared across services).
- R2-009 through R2-013: wire canary mode into each of the 5 services' existing canary-eligible tests (task-store, chat, metrics, admin, agent) — fan-out per the non-negotiable one-service-per-task rule.
- R2-014: add the `smoke`/`canary` CI job split per Phase 2's contract (paired branch-protection task per the repo-config pairing rule — R2-015).

### Milestone 4 — High-tier coverage
**DOD:** 100% of `high` tier inventory items covered.

- Per Phase 1/3, `high` tier coverage is already comprehensive across all services — no net-new `high`-tier test-writing task identified in this pass. The chat-service integration net-new item (once CI wiring lands in M1) is the one item to watch here if/when new chat functionality is added, but there's no existing `high`-tier chat functionality identified as currently uncovered.

### Milestone 5 — Cleanup
**DOD:** Zero tests in `rebuild` bucket. CI green.

- By this point M1's R2-002–R2-005 will have already resolved the 4 rebuild-bucket files — Milestone 5 here is confirmation/verification only (R2-016), not new fix work. This ordering intentionally front-loads the isolation-bug fixes into M1 rather than deferring them to M5, since they're small, high-confidence fixes that don't need M2's new coverage to land first (unlike a true "delete redundant test" cleanup, which does need to wait for its replacement).

## 6. Open risks

- **This roadmap overlaps substantially with an already-published, still-open batch of 42 GitHub issues (#1390–#1431, `T-001`–`T-032`/`P-003`, tracking issue #1431) from a prior pipeline run dated 2026-07-11.** That prior run's source docs (`test-inventory.md`, `test-migration.md`, `test-readiness-plan.md`) never landed on `main` — its context links point at commit `90813fda385aa0ba30f1c9366ad4a564f73c12df`, which is a real `main` commit but an unrelated one (a GHCR build-workflow change), meaning the prior run's own branch/PR was lost before merging (its docs exist nowhere in git history on any reachable ref). The GitHub issues it created are real, live, and still open — none closed, none referenced by any PR yet. This refresh's task list was generated independently from a fresh Phase 1-4 run and, unsurprisingly, converges on much of the same underlying work (chat DB CI wiring, canary rails, naming-convention renames, branch-protection follow-up) but with different granularity and IDs. This refresh's task IDs were deliberately prefixed `R2-`/`R2P-` (instead of reusing `T-`/`P-`) specifically to avoid the Phase 5 dedup mechanism silently matching or colliding with the older, unrelated-content `T-001`–`T-032` range. **A human should reconcile the two batches** — likely by triaging the older batch (close as superseded, or cherry-pick any task not covered here) before both sets sit in the tracker simultaneously and confuse anyone picking up "next ready issue."
- **Whether the 4 isolation-bug failures (R2-002–R2-005) reproduce in real CI is unconfirmed.** This agent's sandbox happens to share exactly the ambient conditions that trigger them (it's itself a Shipwright agent runtime — real marketplace dir, real K8s ServiceAccount CA, real `ANTHROPIC_MODEL`). `.github/workflows/ci.yml`'s runner is a stock GitHub Actions `ubuntu-latest` box and likely does NOT share these conditions, meaning these 4 failures may currently be invisible to CI entirely (passing there, failing only in agent-dogfooding sandboxes) — worth a human confirming via a CI log check before treating R2-002–R2-005 as urgent-priority. Either way the fix is correct and low-risk to land.
- **Branch-protection current state is unverified** (R2-015's verification command needs a human or a properly-scoped token — this agent's GitHub token returned `403 Resource not accessible by personal access token` on `gh api repos/app-vitals/shipwright/branches/main/protection` in both Phase 2 and here). Cannot confirm whether `main` currently requires any status checks at all; the roadmap assumes it will need updating once the canary job (R2-014) lands, per the repo-config pairing rule, but the starting state is unknown.
- **The admin agent-provisioning journey (R2-007) is genuinely destructive** (real K8s Deployment/Secret creation, real Slack app manifest creation) even at the integration layer with injected fakes for the *external* clients — the *test itself* is safe (no real external calls), but designing the composed test requires care that the fake-client call sequence actually matches production ordering (token → Secret → Deployment, DB row last) rather than just individually-correct-but-differently-ordered fakes. This is a design nuance for whoever picks up R2-007, not a blocker.
- **R2-001's rename scope (~15 files) is a judgment call on category (a) vs (b) from Phase 3's Promote/deepen section** — this plan chose (a) (formalize `.content.test.ts` as a fifth category) over (b) (accept as unit with a suffix mismatch) because it's more consistent with this repo's evident naming discipline, but a human/team may reasonably prefer (b) to avoid touching 15 files for a cosmetic gain. Flagging explicitly rather than treating R2-001 as forced.
- **Chat's zero integration-test count (item 3 in The Gap) might be intentional, not a gap** — chat is the newest of the three Prisma-backed services (per its migration timestamps, `20260703000000_init`, notably later than admin's and task-store's earliest migrations) and may simply not have had time to accrue integration tests yet, rather than reflecting an oversight. R2-006a (CI wiring) is worth doing regardless since it's zero-cost today, but R2-006b (the first actual test) shouldn't be treated as "missing" work — it's genuinely net-new, sequenced whenever chat's next DB-touching feature lands.
- **E2E speed/pass status is entirely unverified in this pipeline run** (sandbox has no launchable Chromium — a known, pre-existing, unrelated environment gap). The 52 E2E assertions across 3 files were confirmed to exist and were read for classification purposes, but their current pass/fail state and timing rely entirely on trusting CI's own Playwright run rather than this agent's local verification.
