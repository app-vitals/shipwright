## Canary execution contract

Per Step 4 of `test-design/SKILL.md`, the canary-execution contract
(`${CLAUDE_PLUGIN_ROOT}/skills/canary-execution/SKILL.md`) is included verbatim below.

**Applicability finding (carried forward from Phase 1, re-confirmed this pass):** root
`CLAUDE.md` declares `## Deploy model` as **`direct`** — "ships via Helm chart +
GHCR-pinned image tags, with no staging/production GitHub Environments and no
deploy-staging→canary→promote-to-production pipeline." A canary suite validates a
**post-deploy, pre-promotion** environment (`staging`) before promoting to
`production`; this repo has no such intermediate environment for the contract to gate.
**The canary execution contract therefore does not apply to this repo today.** This
section documents the contract as reference material (so a future move to a staged
deploy model has zero rediscovery cost) and records what changes to make it apply — it
does not prescribe building `test:canary` now.

### The contract, verbatim (for reference)

- **Entry point:** a dedicated `test:canary` npm script or a dedicated Playwright
  project (e.g. `playwright.canary.config.ts`) — must never reuse `test:smoke` / the
  local smoke runner. CI wires two distinct jobs, `smoke` (pre-merge, boots local deps)
  and `canary` (post-deploy, sets `TEST_TARGET_URL`) — they must not share a script.
- **Mode selection:** every canary-eligible smoke/E2E test reads `TEST_TARGET_URL` at
  process start. Unset → local mode (boot deps inline, hit `localhost`). Set → canary
  mode (skip local boot, hit `$TEST_TARGET_URL` directly, canary-only auth via
  `TEST_AUTH_TOKEN`). Test code is identical; only the setup-fixture layer branches.
- **Eligibility rules (all must hold):** (1) layer ∈ {smoke, E2E}; (2) criticality ∈
  {critical, high}; (3) read-only OR self-cleaning (try/finally teardown even on
  assertion failure); (4) no destructive side effects, canary auth scoped to reject even
  attempts; (5) tolerates eventual consistency via retry-with-backoff; (6) zero
  DB-dependent local setup — a test that boots a local DB is never canary-eligible.
- **Budget:** per-test 95p <5s smoke / <30s E2E; **suite wall-time hard cap 60s total**.
- **Gating:** failing in local mode blocks the PR (local mode is the primary contract).
  Failing in canary mode blocks promotion staging → production. Canary is a **deploy
  gate, never a merge gate**.
- **Canary safety lint:** before promoting a test file to the canary suite, grep it for
  write patterns (`.create`/`.update`/`.delete`/`.insert`/`.upsert`/`POST`/`PUT`/`PATCH`/`DELETE`)
  and classify every match as setup-only-with-teardown, canary-scoped, or unsafe. Any
  unsafe match disqualifies the file — see `canary-execution/SKILL.md` for the full
  step-by-step lint.

### What would need to change for this contract to apply here

1. Stand up a `staging` deploy target (or equivalent pre-production environment) with a
   stable `TEST_TARGET_URL` and a scoped `TEST_AUTH_TOKEN` issuance path.
2. Add a promotion step between "deploy to staging" and "deploy to production" for the
   canary job to gate.
3. Only then: define the critical-path roster (Phase 1's "Critical-path roster (canary
   suite definition)" section, currently empty by design), reserve the already-defined
   `.canary.test.ts` suffix (see `docs/test-readiness/naming.md` — reserved but unused),
   and wire the `smoke` / `canary` CI jobs per the contract above.
4. Until then: **no `test:canary` script, no `.canary.test.ts` files, and no `canary`
   CI job should be created.** The `.canary.test.ts` suffix stays reserved-but-unused in
   `bunfig.toml`'s `pathIgnorePatterns` (already the case) as a naming placeholder only.

**Net effect on this design:** Step 4's usual output (a populated canary CI job, a
`test:canary` script name, a critical-path roster) is replaced with the applicability
finding above. Every other step (frameworks, local substitutes, test-data strategy, CI
pipeline shape, coverage, speed budgets, shared helpers, repo configuration) proceeds
unchanged — none of them depend on canary applying.

## Test-data strategy

- **Per-service DBs vs. shared:** per-service, already the case — each Prisma-backed
  service (`admin`, `task-store`, `chat`) reads its own `DATABASE_URL_SHIPWRIGHT_<SERVICE>`
  (root `CLAUDE.md`'s "Database env vars" table); no shared connection string anywhere.
  This matches the modern-microservice-convention default and requires no change.
- **Real DB per test run vs. transaction rollback:** real DB per test run. Each DB
  integration test's `beforeEach` spins up a brand-new `PrismaClient` (its own pooled
  connection, `connection_limit` bounding the pool) and `afterEach` disconnects it —
  already the documented pattern in `ci.yml`'s comments. Transaction-rollback isolation
  was considered and rejected: it requires wrapping the service layer's own transaction
  boundaries in an outer test transaction, which the current service-layer interfaces
  don't expose, and real-DB-per-run at this test volume (per Phase 1: 3 Prisma-backed
  services, not dozens) doesn't yet justify the added complexity. The two new admin
  services this pass (`agent-chat-tokens.ts`, `agent-work-queue.ts`) don't change this
  calculus — same volume tier.
- **Factories vs. fixtures:** both, split by purpose — factories (in-line builder
  functions returning valid entity shapes with overridable fields) for DB integration
  tests needing variability across many test cases; hand-authored fixture JSON (the
  existing `src/fixtures/` convention) for recorded-fixture doubles representing a
  specific external-service response shape (stable golden cases, not meant to vary).
- **Seed strategy:** explicit, named seed sets per test file — no reliance on
  migration-time seed data. `scripts/seed-*.ts` (noted in the inventory as dev-only DB
  seeders) are for local `task stack` development, not test seeding; tests seed their
  own state directly via the service layer in `beforeEach`.
- **API-first seeding:** seed through the public API / service layer wherever it exists
  (e.g. `taskService.create(...)` rather than a raw Prisma `create` call in the test),
  per the SKILL's rule that direct DB writes bypass validations and derived-field logic
  that accumulate over time. Direct DB writes are the explicit fallback only for
  corrupt-state or legacy-data tests that the service layer cannot express (e.g.
  `task-store/src/stale-claim-reaper.ts`'s abandoned-claim scenarios, and any test
  exercising `lib/claim-ttl.ts`'s boundary constants against a claim that's artificially
  aged past `DEFAULT_CLAIM_TTL_MS` — by definition a state the service layer's own API
  wouldn't produce on a fresh write).
- **Eventual consistency:** the inventory found no queue/retry infrastructure in this
  repo (Phase 1, category 4 detail: "no queue/retry infrastructure observed") and no
  async worker/event-bus component. The two async-shaped paths are
  `agent/src/loop-orchestrator.ts`'s poll loop (task-store claim → dispatch → status
  update) and `agent/src/chat-poller.ts`'s poll loop (chat-svc queue → agent poll →
  Claude session → reply) — both are tested by directly awaiting the orchestration
  function under test with an injected `Clock`, not by polling for eventual state,
  since the "worker" in both cases is the function under test itself, not a detached
  background process. No wait/poll-with-backoff pattern is needed anywhere in this
  repo's test suite today.

## CI pipeline shape

```
┌──────────────────────┐
│  changes (path-diff)  │  gates whether app jobs run at all
└──────────┬─────────────┘
           │
   ┌───────┴────────────────────────────────────────┐
   │                                                  │
┌──▼───────────────────────────┐   ┌──────────────────▼───────┐
│ ci: lint → check-strings →   │   │ site: build → brand-lint  │
│ typecheck → check-config-docs│   │  → Playwright smoke       │
│ → unit+integration+smoke     │   │  (site/*.spec.ts)         │
│  (task test:coverage, all    │   └────────────────────────────┘
│  8 workspaces, one combined  │
│  job) → secret-scan          │   ┌────────────────────────────┐
│  [needs real Postgres        │   │ admin-docker-build          │
│   service container:         │   │ agent-docker-build          │
│   admin+task-store+chat]     │   │ metrics-docker-build        │
└──────────┬────────────────────┘   └────────────────────────────┘
           │
           │ (needs: ci)
  ┌────────┴─────────┐
  │                   │
┌─▼──────────┐  ┌─────▼────────┐
│ e2e         │  │ admin-e2e    │   ← Playwright, parallel, post-ci
│ (metrics    │  │ (admin UI)   │
│  dashboard) │  │              │
└─────────────┘  └──────────────┘
           │
     merge → main → (direct deploy; no canary/promotion stage)
```

This is the **existing** `.github/workflows/ci.yml` shape, re-confirmed against the
current file this pass (job list: `changes`, `ci`, `site`, `admin-docker-build`,
`agent-docker-build`, `metrics-docker-build`, `e2e`, `admin-e2e`) — documented here as
the target because it already matches the SKILL's prescription (below) with no changes
needed to the shape itself, only to what's inside two of the boxes (see gaps noted
throughout this doc).

- **Layer order:** unit → integration → smoke, all inside the single `ci` job (fail-fast
  is intra-job: `bun test` stops the whole `task test:coverage` run on first failure by
  default within a package, and `task ci`'s step order means lint/typecheck fail before
  the test step ever runs). E2E (`e2e`, `admin-e2e`) runs `needs: ci` — post-merge-gate,
  pre-merge-CI, i.e. still required to merge but only after the unit/integration/smoke
  combined job is green. Content tests run alongside unit inside the same `bun test`
  invocation — no separate CI stage, matching Step 6's guidance that content has no I/O
  boundary distinct enough to warrant its own job; the 12 new skill-prompt content tests
  this pass slot in the same way with zero pipeline change.
  **Canary does not appear** — see the Canary execution contract section; there is no
  post-deploy stage in a `direct` deploy model for a canary job to occupy.
- **Fail-fast:** a unit failure fails the whole `ci` job before `e2e`/`admin-e2e` run
  (`needs: ci`), matching "integration failures skip smoke" by construction — they're
  the same job, same process, same `bun test` invocation.
- **Parallelism (per-workspace matrix — evaluated against Step 6's override
  conditions):** the `ci` job runs unit+integration+smoke as **one combined job across
  all 8 workspaces** (`task test:coverage` → `bun test --coverage` at the repo root),
  not a per-workspace matrix. Checked against the three override conditions:
  - **(a) Budget miss:** no — current full-suite wall time comfortably fits the <15min
    PR budget (see Speed budgets below). The ~17 new source files this pass add a
    proportionally small number of new tests to reach target coverage; not enough to
    threaten the budget on its own.
  - **(b) Documented flake/slowness history:** two pre-existing full-suite flakes remain
    on record (`metrics/` — ~15 pre-existing test failures reproducing on clean `main`,
    and `admin/` — a 1-test flake that varies which test) but neither is isolated by
    workspace-matrix sharding today, and sharding would not fix either (both are
    order/environment flakes, not slowness). Not a justification for a matrix split.
  - **(c) Overhead-justified parallel savings:** no — with 8 workspaces at the current
    per-workspace test volume, an 8-way matrix's fixed per-job overhead (checkout,
    `bun install`, Postgres container init × 8) still would not pay for itself 2x over
    versus one combined job's single setup cost.

  **Conclusion: keep the single combined `ci` job.** Default to the cheaper shape per
  Step 6 — none of the three override conditions hold. E2E is already split into two
  jobs (`e2e` for metrics, `admin-e2e` for admin) — that split is justified under (b)/(c)
  implicitly (they're different Playwright projects against different running services,
  not a cost-only sharding decision) and is left as-is; it is not a "layer sharded by
  workspace" instance of the pattern Step 6 gates, since each targets a genuinely
  different browser-driven surface.
- **Budget:** <15min total per PR — see Speed budgets below for the measured/target
  breakdown per stage.

### Naming convention and runner-exclusion config

Per Step 6's requirement that a naming convention and its enforcing exclusion config are
inseparable — both already exist in this repo, confirmed against `docs/test-readiness/naming.md`
and `bunfig.toml`:

1. **File-naming convention** (repo's own five-suffix scheme, per CLAUDE.md):
   - `*.unit.test.ts` — unit
   - `*.integration.test.ts` — integration
   - `*.smoke.test.ts` — smoke
   - `*.spec.ts` (site only) — e2e
   - `*.content.test.ts` — content
   - `*.canary.test.ts` — **reserved, unused** (canary contract not applicable — see above)
2. **Runner-exclusion config:**
   - `bun test` (unit/integration/smoke/content) discovers by default root scan; no
     `pathIgnorePatterns` entry needed for these four since they share one runner by
     design (layer is a naming label, not a separate runner invocation, for these four).
   - `bunfig.toml`'s `pathIgnorePatterns` excludes `site/**`, `metrics/e2e/**`,
     `admin/e2e/**` from the root `bun test` scan — Playwright's `*.spec.ts`/`*.e2e.ts`
     files would otherwise crash Bun's runner. Confirmed still the mechanism per root
     CLAUDE.md.
   - Playwright `testMatch` in each of the three `playwright.config.ts` files scopes to
     that surface's own spec files only.
   - Coverage config (`scripts/check-coverage.ts` + Bun's native lcov output) excludes
     test files themselves, fixtures, and generated files (`lib/*-types.ts` OpenAPI
     types, per Phase 1's "generated code — excluded" classification) from the
     line/branch denominator.
3. **Verification:** `bun test --dry-run 2>&1 | grep -v '\.unit\.\|\.content\.'` should
   return only integration/smoke files if run scoped to those two layers; the inverse
   check works per-layer. (Per project memory: `bun test`'s `--filter`/`--dry-run` flags
   are not always reliable in this sandbox — treat as a CI-only verification, not a
   local one.)

## Speed budgets

Per Step 8, the default budget table from `speed-budgets/SKILL.md`, included verbatim —
this repo does not tighten any budget below the default.

| Layer | Per-test 95p | Per-test hard cap | Suite target |
|---|---|---|---|
| Unit | <50 ms | <200 ms | <30 s |
| Integration | <2 s | <10 s | <3 min |
| Smoke | <5 s | <15 s | <30 s sequential / <60 s full |
| E2E | <30 s | <90 s | <10 min |
| Full PR pipeline | — | — | **<15 min** |
| Canary suite | — | — | **<60 s** (not applicable — see Canary execution contract) |

**Parallelization plan:**

- **Unit + integration + smoke (the combined `ci` job):** Bun's test runner
  parallelizes within a single process by default; no explicit worker-count flag is
  set in `task test` / `task test:coverage` today, and none is being newly prescribed —
  the single-job shape (see CI pipeline shape above) means the whole layer stack runs
  as one `bun test` invocation with Bun's own internal concurrency, not a CI-level
  worker matrix. This is sufficient at current suite size to hit the <15min full-PR
  budget with headroom.
- **E2E (`e2e`, `admin-e2e`):** two separate CI jobs, run in parallel with each other
  (both `needs: ci`, no dependency between them) — this is the sharding, and it's
  justified because they exercise genuinely different Playwright projects/servers, not
  a cost-only split of one logical suite. Within each job, Playwright's own
  default worker parallelism applies (no custom `workers:` override observed in
  `metrics/playwright.config.ts` / `admin/playwright.config.ts` / `site/playwright.config.ts`
  — default to CPU-count-based parallelism, which is sufficient at 1-2 spec files per
  surface today).
- **If suite growth later threatens the <15min budget:** re-evaluate the per-workspace
  matrix override conditions in the CI pipeline shape section above — specifically
  condition (a) (budget miss) is the trigger to revisit, not a fixed test-count
  threshold. The ~17-file delta this pass is not, on its own, that trigger.

**Measured baseline (T-067, folding in T-002):** the default budget table above is no
longer just a target — real per-layer numbers now exist in
[`speed-baseline.md`](./speed-baseline.md), measured across all five workspaces (plugin,
metrics dashboard, agent, admin, site). Every measured layer falls comfortably within
budget:

- **Unit:** 1.05s–2.50s per package (well under the <30s suite target)
- **Integration:** 0.15s–5.35s per package (well under the <3 min suite target)
- **Smoke:** 3.24s–4.62s per package (admin, metrics — the only two packages with
  smoke-layer tests today), well under the <30s/<60s suite target
- **E2E:** 4s–40s via CI (metrics dashboard, admin UI, and site), well under the <30s
  per-test 95p / <90s hard cap
- **Full suite:** 34.23s wall-clock (3842 pass, 242 skip, 4084 tests across 198 files),
  well within the <15 min full-PR budget

See `speed-baseline.md` for the full per-package breakdown, methodology, and the exact
verification command output. Future `/test-migration` passes should cite this baseline
rather than re-flagging speed as "not measured."

  No new shared cross-workspace helper needed — auth shapes differ enough per service
  (session-or-bearer for admin, bearer-only for task-store/chat) that a shared builder
  would abstract away meaningful differences.
- **Test-user / test-agent factory** — needed for `admin`'s Agent/Env/Cron/Tool/Token
  CRUD integration tests (including the two new services, `agent-chat-tokens.ts` and

  in-line builder functions (per the factories-vs-fixtures split above) colocated with
  each service's `*.integration.test.ts` files, not a shared cross-workspace package —
  the entity shapes are service-specific (Prisma models differ per schema). Phase 3
  should confirm whether this already exists per-service or still needs building.
- **DB reset helper** — `beforeEach`/`afterEach` PrismaClient connect/disconnect pattern
  already documented in `ci.yml`'s comments (real DB per test run, no rollback). No
  shared truncate-helper needed if each test seeds only what it needs and Prisma's

  cross-test pollution today given 3 separate schemas.
- **Time freezer (`Clock` injection)** — already a hard rule in root `CLAUDE.md`'s Test

  interface. `metrics/src/lib/coalescing-cache.ts` (also new this pass) consumes this same
  injected-`Clock` pattern for its lazy expiry check). No new work — this is the strongest

- **Recorded-fixture loader** — already exists as the `Recorded*Client` pattern
  (`admin/src/kubernetes-client.ts`'s `RecordedKubernetesClient`,
  `metrics/src/providers/task-store-recorded.ts`, `metrics/src/fixtures/task-store-fixtures.ts`

## Repo configuration

Per Step 10 (`repo-config/SKILL.md`). Without this section, the test pipeline above is
advisory — CI can run and still not gate `main`.

### Branch protection (`main`)

- **Required PR** — no direct pushes (assume already true; verify in Phase 4 against the
  live GitHub branch-protection API, not assumed from this doc).
- **Required status checks:** `ci` (lint/typecheck/unit/integration/smoke/secret-scan),
  `site` (build/brand-lint/Playwright), `admin-docker-build`, `agent-docker-build`,
  `metrics-docker-build`, `e2e`, `admin-e2e` — every job in `.github/workflows/ci.yml`
  that isn't itself gated to skip on version-bump-only changes (the `changes` job's
  `app==true` condition is fine to leave as-is; a required check that's skipped due to
  the path filter still reports a passing/neutral state to the merge gate, which is the
  intended fail-open behavior documented in `ci.yml`'s own comments).
- **Require branches up to date before merging:** **no.**
- **Require conversation resolution before merging:** **yes.**
- **Required approving reviews:** **0.**
- **Enforce on admins:** **no.**

`shipwright-deploy` is enabled for this repo, so most PRs are agent-authored and
agent-reviewed, and `deploy.md` Step 4b merges every PR with `gh pr merge --admin` by
design, not as an emergency exception. `--admin` bypasses branch protection wholesale, so
up-to-date, required-reviews, and enforce-admins are all moot for the actual merge path:
GitHub blocks self-APPROVE via the API so a required-review count can never be satisfied
by the pipeline itself, `enforce_admins: true` would either block every automated merge or
require the merging account to be a permanent `bypass_actor` (enforcement in name only),
and "up to date" specifically burns real CI minutes — every merge into main invalidates the
"up to date" status of every other open PR, forcing a rebase and CI re-run on each one just
to become mergeable again, with agent PRs landing in parallel that adds up fast. Conversation
resolution is different: it's a zero-cost metadata check (no CI job to re-run), and it
doesn't block the `--admin` merge path either — enable it as a free safety net for any
manual merge path. These are the standard values for every shipwright repo, one flat
default rather than a per-repo choice, since shipwright repos are moving toward
`shipwright-deploy` across the board; see `repo-config/SKILL.md`'s Branch protection
section for the full reasoning.
- **Canary status check:** not applicable / not included — see Canary execution
  contract section; there is no canary job to include or exclude.
