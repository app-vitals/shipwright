# Test System Design

> Phase 2 of the test-readiness pipeline. The greenfield blueprint for the ideal test system that would satisfy the Phase 1 inventory. Does NOT account for existing tests вҖ” that's Phase 3.
>
> Generated by `/test-design` on 2026-07-13.

## Stack profile (from inventory)

- **Language:** TypeScript (Bun runtime), Astro for the standalone marketing site.
- **Package manager:** Bun workspaces (`lib`, `plugins/shipwright`, `metrics`, `agent`, `admin`, `task-store`, `chat`, `mcp-server`).
- **Primary framework:** Hono (`OpenAPIHono` + Zod) for every HTTP service; Prisma/Postgres for admin, task-store, chat; Playwright for E2E across admin, metrics, site.
- **Per-layer counts (from inventory):** critical-path roster spans 9 items across task-store, chat, metrics, admin, agent, and site. External dependencies named: Postgres (Г—3 services), GitHub API/App, Slack (Bolt + provisioning), Kubernetes API, Claude Agent SDK, Google OAuth, Groq/ElevenLabs (voice).
- **Canary suite candidates:** task-store read routes, chat read routes, metrics auth+summary routes, metrics dashboard journey, admin auth+agents-api/admin-ui read routes, admin loginвҶ’agent-detail journey, agent health endpoint, site pages.

## Framework matrix per layer

| Layer | Framework | Rationale |
|---|---|---|
| Unit | `bun test` | Native to the runtime already in use everywhere in this repo; zero extra tooling, fastest possible startup, matches the rubric's TS/Bun default. |
| Integration | `bun test` + recorded fixture doubles / real Postgres per service | The rubric's testcontainers default is unnecessary here вҖ” three services already have first-party Prisma schemas and CI already provisions real `postgres:16` service containers directly (no testcontainers layer needed since CI itself IS the container orchestrator). Recorded-fixture doubles (`Recorded*Client`) cover every third-party/internal-HTTP boundary (GitHub, Slack, K8s, Claude SDK, Groq/ElevenLabs, cross-service HTTP). |
| Smoke | `bun test` + Hono `app.request()` | Every service in this repo is a Hono app composed via an injectable factory (`createXApp(deps)`); in-process `app.request()` exercises the full middleware + routing pipeline with zero socket/port cost вҖ” the rubric's supertest equivalent, but native to the stack already used. |
| E2E | `@playwright/test` | Already in use for admin, metrics, and site. No reason to introduce a second E2E framework. |

## Local execution architecture

Every external dependency from the inventory, with its local substitute. No test in this system may require a hosted external service.

| Dependency | Local substitute | Setup file |
|---|---|---|
| Postgres (admin) | Real `postgres:16` via CI service container / local Postgres; each integration suite provisions schema via `prisma migrate deploy` and owns its own pooled `PrismaClient` (no ORM-level mocking) | `admin/prisma/schema.prisma`; CI wiring in `.github/workflows/ci.yml` |
| Postgres (task-store) | Same pattern, own schema/DB | `task-store/prisma/schema.prisma` |
| Postgres (chat) | Same pattern, own schema/DB вҖ” **not yet wired in CI** (see Blockers) | `chat/prisma/schema.prisma` |
| GitHub API / GitHub App | `RecordedGithubClient` вҖ” recorded fixture double, injected via DI | plugin/agent test fixtures |
| Slack (Bolt events + provisioning API) | Injected fake `App`/`WebClient` and `RecordedSlackProvisioningClient` | `agent/src/slack.test.ts`-adjacent fixtures |
| Kubernetes API | `fetchFn` injection on `HttpKubernetesClient` вҖ” tests supply a fake `fetch` returning canned `Response` objects, never a global override | `admin/src/kubernetes-client.*.test.ts` |
| Claude Agent SDK | Injected `Tracker`/runner interface, recorded transcript fixtures | `agent/src/claude.*.test.ts` |
| Google OAuth | `RecordedGoogleAuthClient` / injected fake implementing the client interface | `admin/src/google-auth-client.*.test.ts` |
| Groq (transcription) / ElevenLabs (TTS) | Recorded-fixture doubles over the injected HTTP call in `voice.ts` | `agent/src/voice.*.test.ts` |
| Cross-service HTTP (metricsвҶ’task-store, metricsвҶ’admin, adminвҶ’chat, mcp-serverвҶ’task-store) | `Recorded*Client` doubles per client interface, or in-process `app.request()` when both sides are Hono apps in the same test process | per-client fixture files, e.g. `metrics/src/fixtures/task-store-fixtures.ts` |

**Blockers (no clean local substitute):**

- None of the named dependencies lack a viable local substitute. Everything in this repo is already built DI-first (interface + `Http*` production impl + `Recorded*`/fake test impl), so the local-substitute pattern is a **description of the existing convention**, not a new recommendation.
- **Process/soft blocker, not a dependency blocker:** the chat service's Postgres integration tests are not wired into `.github/workflows/ci.yml` (no `DATABASE_URL_SHIPWRIGHT_CHAT_TEST` / schema-sync step exists for chat, unlike admin and task-store). This isn't a "no local substitute" problem вҖ” the substitute (real Postgres service container) is already running in CI for the other two services вҖ” it's a CI-wiring gap. Flagged here because Phase 4's roadmap should include a task to add chat's DB to the CI service container step, matching the existing admin/task-store pattern exactly.

## Canary execution contract

Same code, separate entry point:

- **Canary entry point:** `test:canary` (new script, per-package where applicable) or a dedicated `playwright.canary.config.ts` for the E2E-layer canary items (metrics dashboard, admin loginвҶ’agent-detail, site pages) вҖ” never reuses `test:smoke` / `task e2e`.
- **Unset `TEST_TARGET_URL`** вҶ’ local mode. Boot dependencies inline (Postgres service container / recorded fixtures) and hit `localhost:<port>`.
- **Set `TEST_TARGET_URL`** вҶ’ canary mode. Skip local boot entirely. Make HTTP calls directly to `$TEST_TARGET_URL`, using canary-only auth via `TEST_AUTH_TOKEN` / `TEST_CANARY_API_KEY` (never production credentials, never a token minted for a real agent).

**Eligibility:** layer вҲҲ {smoke, E2E} AND criticality вҲҲ {critical, high} AND read-only OR self-cleaning AND zero DB-dependent setup. Applying this to the Phase 1 critical-path roster, the canary suite consists of exactly the 9 roster items вҖ” all read-only `GET` paths and read-only browser journeys. No claim/mutate/provision/deprovision routes qualify (see Phase 1's explicit exclusion list).

**Budget:** <60s wall time for the full canary suite (9 items: task-store health+list, chat health+list, metrics auth+summary, metrics dashboard journey, admin auth+read-routes, admin loginвҶ’detail journey, agent health, site pages вҖ” comfortably fits within budget at smoke-layer <5s/E2E-layer <30s per-test targets with parallel execution).

**Gate:** canary failure blocks promotion (staging вҶ’ production or equivalent). Canary is never a merge gate вҖ” it runs post-deploy only.

**CI jobs:**
- `smoke` вҖ” pre-merge, runs the smoke portion of `task test:coverage` / a dedicated smoke filter, boots local Postgres service containers + recorded fixtures.
- `canary` вҖ” post-deploy, sets `TEST_TARGET_URL` to the deployed staging/production URL, runs `test:canary`.

See `${CLAUDE_PLUGIN_ROOT}/skills/canary-execution/SKILL.md` for the full contract, including the canary safety lint (grep for write-verb patterns, classify each match as setup-only/canary-scoped/unsafe) that must run before any test is promoted into the canary suite.

## Test-data strategy

- **Per-service DBs vs. shared:** per-service, already the case вҖ” `DATABASE_URL_SHIPWRIGHT_ADMIN`, `DATABASE_URL_SHIPWRIGHT_TASK_STORE`, `DATABASE_URL_SHIPWRIGHT_CHAT` are three independent Postgres databases per `CLAUDE.md`'s env-var-namespacing convention. This is the correct target shape вҖ” no change recommended.
- **Real DB per test run vs. transaction rollback:** real DB per test run, not transaction rollback. Each integration `beforeEach` spins up its own pooled `PrismaClient` and `afterEach` disconnects it (per the existing CI comment in `ci.yml`) вҖ” no shared transaction wrapping. This trades a small amount of per-test setup cost for zero cross-test leakage risk, which is the right tradeoff at this repo's current test volume (~100 integration tests total).
- **Factories vs. fixtures:** fixtures for stable golden cases (recorded fixture doubles for all external/cross-service HTTP вҖ” `metrics/src/fixtures/task-store-fixtures.ts` is the existing pattern to replicate). Lightweight inline builder functions (not a full factory library) for DB row construction in integration tests вҖ” keeps dependencies minimal, matches the repo's existing "no new coupling" architectural rule.
- **Seed strategy:** explicit, named seed sets per test file вҖ” each integration/smoke test creates exactly the rows it needs via the service layer (API-first seeding, see below), not a shared global seed script. `agent/workspace/state` and `*/src/fixtures/*` directories are the existing convention for named fixture sets.
- **API-first seeding:** seed through the service layer's own methods (`TaskService.create()`, `PullRequestService.claim()`, etc.) wherever it exists, not raw Prisma writes вҖ” this exercises the same validation/derived-field logic production traffic does. Direct Prisma writes are an explicit fallback only for legacy/corrupt-state fixtures that the service layer cannot express (e.g., a row with a status value no longer reachable via any service method).
- **Eventual consistency:** no async workers (queues, webhooks, event buses) exist in this repo today вҖ” every write is a synchronous Prisma call within the same request. No retry/backoff pattern is needed for local or smoke tests. The one process with async-like behavior is `stale-claim-reaper.ts` (a `setInterval` background job) вҖ” its integration tests should invoke `reap()` directly (synchronously) rather than waiting on the real interval, sidestepping the need for a poll pattern entirely.

- **Layer order:** unit вҶ’ integration вҶ’ smoke. E2E and canary run post-merge / post-deploy.
- **Fail-fast:** a lower-layer failure skips higher layers for that package; independent packages continue in parallel (a task-store failure doesn't block metrics from finishing its own unit+integration run).
- **Parallelism:** all 8 workspaces' unit+integration jobs run in parallel (8-way). Smoke jobs run in parallel with each other (4-way: task-store/mcp-server, chat, metrics, admin), sequenced after every unit+integration job passes. E2E jobs (3-way: metrics, admin, site) run in parallel with each other post-smoke.
- **Budget:** <15 min total per PR (this repo's own `CLAUDE.md` documents `task ci` as the exact merge-blocking chain today; the per-package parallel split above is what gets a monorepo of this size comfortably under that budget as it grows вҖ” the current flat single-job `bun test` across all packages is the simplest viable shape at today's suite size but will need this split once per-package suite time approaches the full-pipeline budget).

**Naming convention and runner-exclusion config:**

1. **File-naming convention** (already fully adopted in this repo per `CLAUDE.md`'s `## Test conventions`):
   - `*.unit.test.ts` вҖ” unit layer
   - `*.integration.test.ts` вҖ” integration layer
   - `*.smoke.test.ts` вҖ” smoke layer
   - `*.e2e.ts` (root-level workspaces) / `*.spec.ts` (site/) вҖ” E2E layer

2. **Runner-exclusion config:**
   - `bun test` picks up all three `*.test.ts` suffixes by default (no separate include/exclude needed since they're one runner); layer separation for CI purposes uses `bun test --filter <pattern>` scoped to file-suffix glob, e.g. `bun test '**/*.unit.test.ts'` for a unit-only job.
   - `bunfig.toml`'s `[test].pathIgnorePatterns` already excludes `site/**`, `metrics/e2e/**`, `admin/e2e/**` from the Bun runner вҖ” this is exactly the runner-exclusion config the E2E layer needs, already correctly wired.
   - Playwright configs (`metrics/playwright.config.ts`, `admin/playwright.config.ts`, `site/playwright.config.ts`) scope to their own `*.e2e.ts`/`*.spec.ts` glob within their directory вҖ” no cross-contamination risk since Bun and Playwright never scan the same files.
   - Coverage config (`scripts/check-coverage.ts`'s `EXCLUDE_PREFIXES`) already excludes process entrypoints (`*/main.ts`, `*/server.ts`, `entrypoint-main.ts`, `serve.ts`) and dev-only seed scripts from the aggregate coverage gate вҖ” correct, since those paths are exercised in practice (real deploys) rather than under test.

3. **Verification:** `bun test '**/*.integration.test.ts' --dry-run 2>&1 | grep -E '\.(unit|smoke)\.test\.ts'` should return nothing вҖ” confirms the integration filter picks up only integration files. Equivalent checks for `.unit.` and `.smoke.` filters. For Playwright: `bunx playwright test --list 2>&1 | grep -v -E '\.e2e\.ts|\.spec\.ts'` should return nothing.

## Coverage targets

| Tier | Layer | Target |
|---|---|---|
| critical | unit | вүҘ95% line, вүҘ95% branch |
| critical | integration | every named boundary |
| critical | smoke | every public route 200/4xx |
| critical | E2E | every top-5 journey |
| high | unit | вүҘ80% line |
| high | integration | every named boundary |
| high | smoke | happy path |
| medium | smoke | spot-check |

**CI enforcement uses a single floor, not per-tier gates:** вүҘ80% line and вүҘ80% function, aggregate-weighted across all counted files вҖ” this is already implemented exactly as prescribed (`scripts/check-coverage.ts`, invoked by `task test:coverage`). Per-tier targets above drive *prioritization* during roadmap planning (Phase 4 writes critical-tier tests first), not a second CI gate вҖ” introducing a criticality-aware CI threshold would require maintaining a criticality-to-file mapping that decays as the codebase evolves, which this design deliberately avoids per the rubric's rationale.

## Speed budgets

| Layer | Per-test 95p | Per-test hard cap | Suite target |
|---|---|---|---|
| Unit | <50 ms | <200 ms | <30 s |
| Integration | <2 s | <10 s | <3 min |
| Smoke | <5 s | <15 s | <30 s seq / <60 s full |
| E2E | <30 s | <90 s | <10 min |
| Full PR pipeline | вҖ” | вҖ” | **<15 min** |
| Canary suite | вҖ” | вҖ” | **<60 s** |

No budget is tightened below the `speed-budgets` skill defaults вҖ” this repo's suite volume (roughly 100 unit, 50 integration, 24 smoke tests today per the file-count sample) comfortably fits the default targets without needing repo-specific tightening.

**Parallelization plan:** 8-way parallel at the unit+integration layer (one job per Bun workspace: `lib`, `plugins/shipwright`, `metrics`, `agent`, `admin`, `task-store`, `chat`, `mcp-server`) вҖ” each workspace's suite is small enough (6-41 source files) that a single worker per workspace hits the target without needing intra-workspace sharding. 4-way parallel at the smoke layer (task-store+mcp-server combined, chat, metrics, admin вҖ” the four services with an HTTP surface). 3-way parallel at the E2E layer (metrics, admin, site вҖ” each already has its own Playwright config and can run as an independent CI job). This matches the CI pipeline diagram above.

## Shared helpers / utilities to build

Most of these already exist in this repo and should be treated as the pattern to keep, not new build items:

- **`Clock` interface + `FixedClock(t)` / `SystemClock`** вҖ” already the universal time-injection pattern (`clock.ts` present in `task-store/`, `chat/`, `agent/`, `admin/`, `metrics/src/lib/`, `plugins/shipwright/scripts/`). No new work; keep enforcing via the isolation contract below.
- **`Recorded*Client` fixture doubles** вҖ” already the pattern for every external/cross-service HTTP boundary (`RecordedTaskStoreClient`/`RecordedAdminMetricsClient` in `metrics/src/providers/task-store-recorded.ts` is the reference implementation other services' doubles should mirror).
- **Injected-`fetchFn` HTTP clients** вҖ” already the pattern for avoiding `global.fetch` overrides (`kubernetes-client.ts`, `tool-caller.ts`, `check-helpers.ts`'s `createTaskStoreClient`).
- **DB reset helper** вҖ” not a shared truncate/rollback helper; the existing per-test `beforeEach`-provisions/`afterEach`-disconnects `PrismaClient` pattern is the convention. No new shared helper needed вҖ” each service's integration test file owns its own setup.
- **HTTP request builders (auth tokens, default headers)** вҖ” recommend extracting a small shared `buildAuthHeader(token)` / `buildBearerRequest(path, token, init)` helper per service's smoke-test file, if not already present, to reduce boilerplate across the growing `*.smoke.test.ts` files. - **Test-user / test-agent factory** вҖ” a lightweight builder function per service (e.g., `buildTestAgent(overrides)`) for admin's Prisma fixtures, to avoid repeating full row literals across integration tests. Recommend as a Phase 4 roadmap item if not already present in `admin/src/fixtures/`.
- **Recorded-fixture loader** вҖ” the `*-fixtures.ts` pattern (`metrics/src/fixtures/task-store-fixtures.ts`, `agent/src/fixtures/`, `admin/src/fixtures/`) is already the convention; no new loader abstraction needed.

## Test isolation contract (carried forward, unchanged)

These hard rules are already correctly encoded in `CLAUDE.md` and should remain unchanged by this refresh:

- **Time:** any production code path calling `new Date()`/`Date.now()` non-trivially must accept a `Clock` interface; tests inject `FixedClock(t)`.
- **External HTTP:** external service clients are defined as interfaces with an `Http*Client` production implementation and a `Recorded*Client` test double вҖ” never `mock.module()`, never `global.fetch`/`global.*` overrides (Bun shares the test process; leaked globals break sibling suites).
- **No global state:** tests must not mutate module-level globals or rely on execution order.
